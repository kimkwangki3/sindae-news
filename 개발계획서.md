## 1. 기술 스택
- **프론트/풀스택**: Next.js 14+ (App Router) + TypeScript
- **스타일**: Tailwind CSS (시안의 디자인 토큰 이식)
- **백엔드/DB/인증/스토리지**: Supabase (PostgreSQL + Auth + Storage)
- **로그인**: 카카오 OAuth (Supabase Auth)
- **배포**: Vercel (지금은 로컬 `npm run dev` 위주)
- **상태/데이터**: 서버 컴포넌트 우선 + `@supabase/ssr`, 필요한 곳만 클라이언트 컴포넌트

## 5. 인증 · 권한 (★ 핵심)
`db/회원등급_설계.md` 기준 3층 모델:
1. **계정 등급** `profiles.role` = guest/member/reporter/admin/superadmin
2. **소속** 업체(1인 1개)·단체(다중, 역할 owner/staff/member)
3. **상태** 정상/정지(is_suspended)/탈퇴(deleted_at)

- `lib/permissions.ts` 에 `can(user, action, target)` 구현 (문서의 의사코드).
- DB는 **RLS로 이중 방어**: 본인 글만 수정, 발행글만 공개 등. `schema.sql`의 정책 패턴을 모든 테이블에 확장.
- 로그인 흐름: 카카오 → 최초 1회 `onboarding`(닉네임 중복확인·설정) → `profiles` 생성.

## 6. 스토리지 버킷
`articles` `board` `market` `business` `org` `ads` — 이미지 업로드. 공개 읽기, 작성자/관리자 쓰기.

## 7. 기능 필수 규칙 (놓치면 안 되는 것)
- 좋아요/싫어요: **IP당 1회**(`article_reactions` unique(article_id, ip_hash)), 변경은 update.
- 댓글·게시판·나눔·홍보·단체글: **로그인 필수**.
- 승인 흐름: 업체·단체·기자·광고·홍보글은 `pending`→관리자(또는 단체운영진) 승인 후 노출.
- **업체 1인 1개** (DB 부분 유니크).
- 단체 가입: 신청 → 해당 단체 **운영진**이 승인(관리자 아님).
- 신고: 전 콘텐츠 → `reports(target_type,target_id)` 통합, 관리자 처리.
- 나눔마켓 목록: **3번째 글마다 인피드 광고** 노출.
- 핫소식: `article_views` 기간 집계(일/주/월).
- 기자: 작성 기사 **법적 책임은 기자 본인** — 서약 동의·서명·일시·IP 저장.
- 통계: `page_views`(방문/PV/유입), `article_views.scroll_pct`(읽음률)·`dwell_ms`(체류).
- 광고: 자동(애드핏/애드센스) + 수동 배너(슬롯별 이미지·링크 교체).
- 법적 페이지: 발행인·편집인 / 청소년보호정책 / 윤리강령 / 정정보도 / 개인정보처리방침.

## 9. 개발 단계 (Phase)
- **Phase 0 — 셋업**: Next.js+TS+Tailwind 생성, Supabase 프로젝트·`schema.sql` 실행, 카카오 OAuth, `.env`, 디자인 토큰 등록, 공통 레이아웃/하단탭/상단 아이덴티티 바.
- **Phase 1 — 기사**: 카테고리·목록·상세, 좋아요(IP)·댓글, 핫소식, 조회/읽음 집계.
- **Phase 2 — 인증·권한**: 카카오 로그인·온보딩(닉네임), `profiles`, permissions+RLS, 마이페이지.
- **Phase 3 — 관리자**: 기사 CRUD·발행, 댓글/신고 처리, 회원관리, 통계 대시보드.
- **Phase 4 — 커뮤니티**: 나눔마켓, 자유게시판(이미지), 제보. 신고 공통화.
- **Phase 5 — 상권·단체**: 업체 등록·승인·홍보글·가게상세, 지역단체 등록·소개·소식·가입(운영진 승인), 카톡채널 연결.
- **Phase 6 — 광고·법적·SEO·배포**: 광고 슬롯/신청, 법적 페이지, 메타·OG·sitemap, Vercel 배포·도메인.
