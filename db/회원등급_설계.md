# 신대신문 — 회원 등급 설계 (최적화)

## 핵심 아이디어: 등급을 "하나의 값"이 아니라 3개 층으로 분리
한 사람이 "일반회원 + 업체 운영 + 단체 2곳 가입 + 기자"처럼 여러 신분을 동시에 가질 수 있다.
그래서 등급을 단일 컬럼으로 두면 안 되고, 아래 3층으로 나눈다.

```
① 계정 등급 (단 1개)   →  profiles.role
② 소속/속성 (0~N개)    →  업체 멤버십 · 단체 멤버십
③ 상태 (1개)           →  정상 / 정지 / 탈퇴
```

---

## ① 계정 등급 (account role) — 한 사람당 하나
| 등급 | 설명 | 부여 |
|---|---|---|
| `guest` | 비로그인. 읽기만 | — |
| `member` | 일반 회원(기본). 댓글·게시판·나눔·신고·가입신청 | 카카오 로그인 시 자동 |
| `reporter` | 시민기자. + 기사 작성·발행 | 기자신청+서약 → 관리자 승인 |
| `admin` | 운영관리자. 콘텐츠·신고·승인 관리 | 지정 |
| `superadmin` | 최고관리자. 관리자 임명·사이트 설정 | 지정 |

> 업체·단체는 계정 등급이 **아니다**. 아래 ②로 처리한다. (그래야 "기자이면서 업체 운영" 같은 조합 가능)

---

## ② 소속 / 속성 — 한 사람당 여러 개 가능
### 업체 (1인 1개만)
- `businesses.owner_id` = 본인. **한 사람당 업체는 0개 또는 1개** (DB 제약으로 강제).
- 승인되면 그 업체의 홍보글 작성 가능.
- 표시: 🏪 업체명 (상단 닉네임 바)

### 지역단체 (1인 N개 가능)
- `org_members` 로 여러 단체에 동시 소속 가능.
- 단체 내 역할: `owner`(대표) / `staff`(운영진) / `member`(일반회원)
- 가입 상태: `pending`(신청) → `approved`(회원) — 단체 운영진이 승인
- 운영진(owner/staff)만 단체 소식 작성·가입 승인 가능.
- 표시: 🏛 단체명 (가입한 단체 모두, 여러 개면 나열)

#### 케이스가 전부 자연스럽게 처리됨
- 단체 미가입 일반회원 → `org_members` 행 없음
- 한 단체 가입 → 행 1개(member)
- 여러 단체 가입 → 행 N개
- 어떤 단체는 운영진, 어떤 단체는 일반회원 → 행마다 role 다름

---

## ③ 상태 (state)
| 상태 | 의미 | 동작 |
|---|---|---|
| 정상 | 기본 | 모든 권한 정상 |
| **정지(차단)** | `is_suspended=true` | 로그인은 되나 **글·댓글·반응·신고 작성 차단**, 읽기만 가능. 사유·기간 기록 |
| 탈퇴 | `deleted_at` 설정 | 비활성. 작성물은 "(탈퇴한 회원)"으로 표기 |

> 정지는 계정 등급과 무관하게 **덮어쓴다**. 기자라도 정지되면 기사 작성 불가.

## 권한 매트릭스 (요약)
| 기능 | guest | member | reporter | 업체owner | 단체staff | admin |
|---|:--:|:--:|:--:|:--:|:--:|:--:|
| 기사·게시판 읽기 | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ |
| 댓글·게시판·나눔·신고 | ❌ | ✅ | ✅ | ✅ | ✅ | ✅ |
| 기사 작성·발행 | ❌ | ❌ | ✅(본인) | ❌ | ❌ | ✅ |
| 업체 홍보글 | ❌ | ❌ | ❌ | ✅(승인업체) | ❌ | ✅ |
| 단체 소식·가입승인 | ❌ | ❌ | ❌ | ❌ | ✅(해당단체) | ✅ |
| 전체 관리·승인·신고처리 | ❌ | ❌ | ❌ | ❌ | ❌ | ✅ |

정지 상태면 위 표의 ✅(읽기 제외)가 모두 ❌로 바뀐다.

## 판정 로직 (의사코드)
```ts
function can(user, action, target) {
  if (user.is_suspended && action !== 'read') return false;
  if (user.role === 'admin' || user.role === 'superadmin') return true;

  switch (action) {
    case 'write_article':  return user.role === 'reporter' && target.author_id === user.id;
    case 'write_promo':    return ownsApprovedBusiness(user, target.business_id);
    case 'write_org_post': return isOrgStaff(user, target.org_id);
    case 'approve_member': return isOrgStaff(user, target.org_id);
    case 'comment':
    case 'write_board':
    case 'write_market':   return user.role !== 'guest';
    default:               return false;
  }
}
```

## DB로 강제하는 핵심 제약
- 업체 1인 1개 → `businesses(owner_id)` 부분 유니크(승인·대기 상태 한정)
- 단체 다중 가입 → `org_members(org_id, user_id)` 유니크(같은 단체 중복가입만 방지)
- 닉네임 중복 불가 → `profiles.nickname` UNIQUE
- 좋아요 IP/유저 1회 → 반응 테이블 유니크
